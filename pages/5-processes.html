5. Process Management

Controlling running programs, signals, and background jobs.

1. Concept Overview

Every program running on Linux is a Process. The kernel assigns each process a unique ID called a PID.

Processes have parents (who started them), owners (who ran them), and states (Running, Sleeping, Zombie). You can control processes by sending them Signals (like "Stop", "Terminate", or "Kill").

4. Deep Dive: Signals

  

    "Killing" a process actually means sending it a signal.
  

  
    
    
  • SIGTERM (15): "Please stop." The default. Allows the app to save data and close files.
    • 
    
  • SIGKILL (9): "Die immediately." The kernel removes the process instantly. Data may be lost.
    
    • 
    
  • SIGINT (2): "Interrupt." Sent when you press Ctrl+C.
    • 
  






  
5. Common Pitfalls

  
    
    
  • Zombie Processes: A "Zombie" (defunct) is a process that has finished but its parent hasn't
      checked on it yet. You can't kill a zombie (it's already dead!). You must kill its parent to
      clean it up.
    • 
    
  • Killing PID 1: PID 1 is the init system (systemd). If you kill it, the system crashes (kernel
      panic).
    •

    4. 
    
  4. Who owns the CPU: Use ps to list the top 5 CPU-consuming processes.

  5. Port conflict debugging: If a service won’t start because a port is in use, identify the owning process.

  6. Kill unresponsive session: Find the PID of your current terminal session and kill it to force a logout.

  7. Run process after logout: Start a long-running script (e.g. sleep 1000) and detach it from the terminal so it survives shell closure.
